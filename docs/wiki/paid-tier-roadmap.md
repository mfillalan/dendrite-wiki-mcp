# Paid Tier Roadmap

This page is the active build tracker for paid product tiers (Pro Individual, Team, Services). It complements [Commercialization Plan](./commercialization-plan.md) (the strategy) and [Release Readiness Roadmap](./release-readiness-roadmap.md) (the launch sequencing).

The decision driving this page: rather than wait for paid customers before building paid features, the operator is building Pro-tier capabilities now as enhanced free product polish, then introducing gating only when there is a customer to gate for. This is the "build the value first, monetize later" pattern.

Specific pricing, per-feature free-vs-paid splits, and license-gating decisions live in private operator notes rather than this public page. This page tracks build order and feature scope only.

## Build Order Decision

The order is intentional and follows the principle that each layer must justify the next:

1. **Pro features as enhanced free polish first.** Build the richer reports, exportable artifacts, and comparison views as part of the free product. They improve adoption by themselves and become the things to consider gating later.
2. **Services revenue second.** Setup sessions and workflow audits require zero new code. They generate revenue, validate willingness to pay, and surface what teams actually need.
3. **License-key gating third.** Only when a customer wants to pay for a specific feature, add the gating mechanism around that feature. Avoid building DRM before product-market fit.
4. **Team tier last.** Hosted dashboards and multi-user coordination are real engineering investment. Defer until a team has explicitly asked for them and has paid for setup.

## Pro Tier Feature Track

Pro is "make the local product feel premium for one developer." Each feature here is buildable today against existing data with no hosted infrastructure.

### P1: Exportable Benchmark Report

**What it is:** a single self-contained HTML file that renders the local benchmark history (wiki health metrics, recall quality, automatic event summary) into a sharable report. Generated by `dendrite-wiki report:export`.

**Why it matters first:** the commercialization plan calls for a "Before Dendrite / After Dendrite" example project story. An exportable report is exactly that artifact — proof a beta user can hand to their team to justify adoption.

**Build acceptance:**
- New `report:export` CLI command writes a single HTML file (no external dependencies, inline CSS, embedded data).
- The HTML renders the same baseline-vs-latest comparison the in-browser BenchmarkReport.vue shows.
- Includes a clearly attributed "Generated by Dendrite Wiki MCP" footer with version.
- Default output path is `docs/public/benchmark-report.html` (gitignored), overrideable with `--output`.
- Test fixture proves the HTML contains key metrics from a real snapshot.

### P2: `dendrite-wiki doctor` Command

**What it is:** a project-health audit command that checks for: missing client config files, missing seed wiki pages, stale benchmark (>14 days), absent project-log entries this week, lint findings, contradictory memories, drift between client configs.

**Why it matters:** the most common beta-user friction will be silent setup problems. A `doctor` command turns "I don't know why this isn't working" into a numbered fix list.

**Build acceptance:**
- `dendrite-wiki doctor` runs all checks and prints a prioritized report.
- Exit code 1 when any critical issue is found, 0 when clean.
- Each finding includes a concrete fix command.
- `--fix` flag auto-applies safe fixes (regenerating seed pages, etc.).

### P3: Multi-Snapshot Comparison View

**What it is:** an extension to the in-browser BenchmarkReport.vue that lets the operator pick any two snapshots (not just first-vs-latest) and see a delta. Today the report only shows baseline-vs-latest.

**Why it matters:** answers "how did this week compare to last week?" without manual JSON inspection. Surfaces trend regressions that get hidden by the baseline drift.

**Build acceptance:**
- Snapshot picker (two select dropdowns, A and B).
- Side-by-side metric delta with up/down indicators.
- URL state (so a comparison can be shared as a link to the local docs server).

### P4: Branded Report Templates

**What it is:** the exportable HTML report (P1) gains a `--theme` flag and supports `--logo path/to/logo.png` and `--brand-color #hex`. Lets a consultant produce a report with their own branding when delivering to a client.

**Why it matters:** turns the export into a deliverable. A consultant doing a setup session can hand off a branded report instead of a plain Dendrite-branded one.

**Build acceptance:**
- `--theme`, `--logo`, `--brand-color`, `--report-title` CLI flags.
- Logo is base64-embedded into the HTML to keep the file self-contained.
- A built-in "minimal" theme alongside the default.

### P5: Historical Trend Sparklines on Each Wiki Page

**What it is:** each wiki page in the browser gets a small sparkline showing its claim count, source coverage, and recall hit count over time.

**Why it matters:** today the benchmark is project-wide. Page-level trends would let the operator say "the auth-flow page has been losing claims for 3 weeks" without manually grepping history.

**Build acceptance:**
- VitePress component injected into wiki page layout.
- Sparkline data sourced from existing snapshot history (no new collection needed).
- Hover tooltip with snapshot timestamp + value.

## Team Tier Feature Track

Team is "coordination across multiple developers." This requires real new infrastructure and is deferred until at least one team has paid for setup or asked for it.

### T1: Install Profile Packs

**What it is:** a JSON or YAML format that captures a team's "this is how we install Dendrite" — which client profile, which seed pages to override, which guidance file edits to apply. `dendrite-wiki init --pack my-team-pack.json` applies the whole thing.

**Why it matters:** onboarding a new developer to a team's Dendrite setup is currently "follow these manual steps." Packs make it one command.

**Status:** not started. Trigger to start: any team asking how to standardize setup.

### T2: Hosted Aggregate Dashboard

**What it is:** a Supabase-backed Next.js dashboard that ingests the existing opt-in telemetry payload across team members and shows team-wide trends. The schema and ingestion already exist — what's missing is the dashboard UI.

**Why it matters:** the visible team-tier value. An engineering manager can see "my team's wiki coverage went from 40% to 75% this quarter."

**Status:** not started. Estimated 2–4 weeks of focused work when justified.

### T3: License-Key Gate System

**What it is:** a lightweight license-check mechanism. Free features run unconditionally. Pro/Team features check for a license key (env var, config file, or hosted endpoint). License checks must be best-effort with generous offline grace — no hard fail without internet access.

**Why it matters:** required infrastructure for any paid feature gating. Currently nothing is gated because nothing is paid.

**Status:** not started. Trigger: at least one Pro feature with paying customer demand.

### T4: Support SLA Workflow

**What it is:** a documented process for how Team customers report issues, what response times to expect, and how escalation works. This is operational policy, not code.

**Status:** not started. Trigger: first Team contract.

## Services Track

Services need zero new code and can begin immediately once the free product is publicly released. Three offerings are planned: setup sessions, workflow audits, and custom integration work. Scoping, pricing, and engagement format live in private operator notes; this page tracks only that the offerings exist and depend on the free product being publicly available.

## Why No Gating Yet

The current approach is to build Pro-tier features as **enhanced free capabilities** until a paying customer creates the need for gating. Reasons:

- Building DRM before product-market fit wastes engineering time on infrastructure that may never serve revenue.
- Free-tier polish drives adoption, which drives the qualified leads needed for Services revenue.
- Each Pro feature is independently useful as part of the free product; the gating layer is a strict superset that can be added later without redesigning the feature.
- The repository is local-first by design; aggressive license enforcement would betray the trust posture documented in [Product Vision](./product-vision.md).

When the first paying customer arrives, gating decisions can be made per-feature based on actual willingness to pay rather than speculation.
